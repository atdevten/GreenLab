BACKEND DESIGN PRINCIPLES
Domain Ownership
Each service owns exactly one domain and one database. No shared databases between services. Cross-domain reads go through the owning service's API.
Async by Default
All high-volume paths (sensor write, alerts, predictions) use Kafka for async decoupling. Sync gRPC only for low-latency user-facing reads.
Idempotent Writes
Every Kafka consumer is idempotent. Re-processing a message produces the same result. Enables safe replay on failure with no duplicate data.
Thin API Gateway
API Gateway handles only auth, rate limiting, and routing. No business logic. Business rules live inside the owning microservice.
Separate Read/Write Models
Write path (Kafka → Flink → InfluxDB) is fully decoupled from read path (Query Service → InfluxDB/Cassandra). Each optimized independently.
Config as Data
Channel schema, alert rules, model assignments are stored data in Config/Channel Service, not code. Users change behavior at runtime without redeployment.
